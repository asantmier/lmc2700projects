Signal Lost

Roguelike action game in Unity

 Signal Lost is the name of a 2D top-down roguelike action game developed as part of a senior capstone project at my highschool. I developed it alongside three of my friends under the mentorship of one of our computer science teachers who had previous corporate project management experience. It was developed from scratch in a one year span.

 During this project, we had no appointed leader, although I often took the role when necessary; however my proper role on the team was simulation code and code organization. Every week our mentor had us report to him our progress, what we still had to do, what we failed to do, and a demonstration of our project. He helped teach us how to give concise and meaningful progress reports as would be expected of a team on a professional project. Additionally, he taught us proper timeline planning and helped us improve the reasonability of our expectations.

 This project was the first time I had tried developing a full scale video game, my first true experience developing with Unity, and my first time programming in a group environment. I learned the importance of balancing code quality with production speed and to develop good systems early rather than rewriting bad ones later. I also learned how to correctly (and incorrectly) use GitHub in a shared repository. I think my most important takeaway from this project was learning how easy it is to underestimate how difficult a task is and to always be conservative in planning because there will always be unforseen delays.

Dungeon Crawler

Roguelike hack n' slash in Java

 As part of a semester long school project in college, I worked with a team of four other students to develop a 2D top-down roguelike hack n' slash game using Java. The style of game and timetable for development were decided by the professor, but the specifics of implementation were left to us.

 This project was, at least to me, much the same as Signal Lost - one of my teammates from that project was actually in my team for this project too! I had already learned many lessons from my previous project experience, so it's harder to say what I learned, though I'm quite certain this project helped solidify the lessons of the past. One new thing I learned is the value of dividing work among a team. I did most of the level generation and physics simulation logic, but left the visual and gameplay elements mostly to my teammates except when they needed help. I learned to appreciate such division of work in which everyone is doing what they are best at rather than everyone needing to develop skills for all parts of development.

 Another unique part of this project was our use of Java. I personally dislike programming visual programs in Java and the specific libraries we were using were not the least bit user friendly, but we managed to make a reasonably good product nevertheless. We also used JUnits to run tests on our code. This was my first time using a testing suite, so I learned quite a lot including how useful automatic tests are. Though I have not yet had the chance to try testing in another project, I'm excited to see what kind of useful building and validation features I can make going forward!

Exodus II Python Library

Python library for manipulating Exodus files

 As part of a year-long junior design class, I am working with four other Georgia Tech students to aid Sandia National Labs in updating their codebase to the modern era. Sandia uses a file format called Exodus II in their finite element analysis applications, but the libraries for directly interacting with Exodus files are rather aged. For a little background: finite element analysis (FEA) is a technique for simulating and analyzing how an engineered object will react to forces applied to it. FEA is a multi-step - and often multi-computer - process so data about the model, simulation, and results are stored in an Exodus file. Analysts usually use software to interact with Exodus files, but when the software can't do what they want, they write scripts to manually modify the files.

 Here's where this project comes into play. Analysts usually want to program in Python or sometimes MATLAB, but unfortunately that requires installing Python, C, and Fortran on every computer that needs to interact with Exodus. Decades old code really has its drawbacks...

 We are developing a standalone Python library that interacts with Exodus files through their underlying netCDF interface. This is a work in progress, so we haven't done much of note yet, but I imagine this will be my most challenging work yet.
